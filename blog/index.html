Building a Production-Ready AI Financial Advisor

A deep dive into how I designed and built FinBot — a domain-specific chatbot powered by Google Gemini with enterprise-grade security, conversation memory, and clean architecture.

📅 February 23, 2026 ⏱️ 8 min read 🏷️ Python, GenAI, Gemini

Large Language Models are powerful, but wrapping one in a production application requires much more than a simple API call. You need security layers, conversation memory, prompt engineering, and clean architecture to build something truly reliable.

In this post, I'll walk you through FinBot — an AI-powered financial advisor chatbot that I built from scratch using Python, Streamlit, and Google's Gemini API. It's designed to be production-ready from day one.

⚠️ Disclaimer: FinBot provides educational financial information only. It is not a substitute for professional financial advice.

💡 Why Build This?

Most LLM chatbot tutorials stop at the "hello world" stage — a single API call with no error handling, no security, and no context management. I wanted to build something that demonstrates real-world engineering practices:

  • Domain Expertise: The bot stays focused on finance — it won't help you write poetry
  • Security: Prompt injection defense, rate limiting, input sanitization
  • Memory: Token-aware conversation history that doesn't blow up API costs
  • Clean Code: Layered architecture with full test coverage

The Message Pipeline

Every user message goes through a 6-step pipeline inside ChatService.process_message():

  1. Sanitize Input — strip control chars, limit length, check for injection
  2. Rate Limit Check — sliding window per session
  3. Build Prompt — detect financial topics, assemble system instructions
  4. Retrieve Memory — get token-trimmed conversation history
  5. Call Gemini API — with retry + exponential backoff
  6. Update Memory — store user and model messages
chat_service.pyPython
def process_message(self, user_input: str) -> str:
    # Step 1: Sanitize input
    sanitized = sanitize_input(user_input)

    # Step 2: Rate limit check
    rate_limiter.check_rate_limit(self.session_id)

    # Step 3: Build prompt
    prompt_result = self._prompt_builder.build(sanitized)

    # Step 4: Get conversation history
    history = self.memory.get_history()

    # Step 5: Call Gemini API
    response = self._client.generate(
        user_message=sanitized,
        system_prompt=prompt_result.system_prompt,
        chat_history=history,
    )

    # Step 6: Update memory
    self.memory.add_user_message(sanitized)
    self.memory.add_model_message(response)
    return response

🛡️ Security: The Hard Part

Finance is a high-stakes domain. A prompt injection attack could trick the bot into giving specific investment advice. I built three layers of defense:

🚫

Prompt Injection Defense

Regex patterns detect "ignore previous instructions", system prompt overrides, and role-switching attempts

🧹

Input Sanitization

Control character removal, length limits (2000 chars), and whitespace normalization

⏱️

Rate Limiting

Sliding window rate limiter — 10 requests per minute per session to prevent API cost abuse

💭 Token-Aware Conversation Memory

LLMs have context windows, and API calls are billed per token. The memory system tracks conversation history while staying within a configurable token budget (default: 4000 tokens). When the budget is exceeded, the oldest turns are trimmed first — keeping recent context fresh.

🤖 Production API Client

The Gemini client wraps the Google GenAI SDK with production concerns:

  • Retry with backoff — 3 attempts with exponential wait (2s → 30s) via tenacity
  • Token usage logging — monitors prompt/response/total tokens per call
  • Quota handling — detects 429 errors and shows user-friendly messages with recovery steps
  • Safety fallbacks — handles empty responses and safety-blocked content gracefully

📚 Key Takeaways

  1. Security first in finance. Prompt injection defense isn't optional — one bypassed guardrail could expose you to real liability.
  2. Memory management is underrated. Token-aware trimming keeps conversations coherent without blowing up costs.
  3. Layered architecture pays off. Separating UI → Service → API made testing trivial and changes low-risk.
  4. Domain prompts > generic prompts. Topic-specific system instructions dramatically improve response quality for finance.
